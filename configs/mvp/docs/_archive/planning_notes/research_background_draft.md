# 项目背景与研究目标

## 1. 我们在研究什么

我们研究的是：**Agentic AI 系统的安全评测问题**，更准确地说，是研究如何构建一个**面向失效归因（attribution-oriented）的 Agent Safety Benchmark**。

这里的“Agent”不是传统只能被动对话的语言模型，而是具备更强行动能力的系统。它通常能够：

- 接收用户目标
- 进行多步规划与推理
- 调用外部工具、API、浏览器、文件系统
- 读取环境反馈（observation）
- 根据新信息持续更新计划并执行动作

也就是说，Agent 不只是“生成文本”，而是在一个闭环中不断地**感知—推理—规划—行动**。  
正因如此，Agent 的安全问题已经不再只是“生成了不该生成的内容”，而是会进一步演化为：

- 执行不安全动作
- 泄露敏感信息
- 被外部环境劫持
- 误信恶意工具输出
- 在无攻击条件下因自身缺陷造成破坏
- 在多步执行中将局部小错放大为系统级风险

因此，我们要研究的核心问题不是单纯的“模型是否会被越狱”，而是：

> **在真实 Agent 场景中，一个系统为什么会失败，它的风险是如何进入、传播并最终造成后果的，我们又该如何系统地评测这种失败。**

---

## 2. 为什么这个问题重要

随着大模型从“聊天机器人”逐渐演化为“可行动的 Agent”，安全风险的形态正在发生根本变化。

在传统 LLM 场景下，安全评测主要关注：
- 是否输出违禁内容
- 是否响应恶意请求
- 是否能抵御 jailbreak

这类问题当然仍然重要，但对于 Agent 来说，这只是很小的一部分。  
Agent 一旦具备工具调用与环境交互能力，风险会扩展到更复杂的层面，例如：

- 在整理文件时误删关键数据
- 在性能优化时关闭监控或安全机制
- 在总结网页时被网页中的隐藏指令劫持
- 在调用第三方工具时将用户敏感数据原样泄露
- 在读取 API 或终端输出时，把恶意文本误当成可信命令
- 在多步规划中基于错误假设持续行动，最终造成严重破坏
- 在多 Agent 协作中被一个恶意子 Agent 逐步污染决策链

这些问题说明，Agent 安全不是单点问题，而是一个**系统安全问题**。  
它涉及：

- 用户输入
- 外部观察
- 工具返回
- 内部策略
- 规划模块
- 执行模块
- 权限控制
- 数据边界
- 多智能体协作关系

因此，如果我们仍然沿用面向单轮文本输出的评测方式，就无法真正理解 Agent 的风险。

---

## 3. 现有评测的主要不足

尽管近年来已经出现大量与 Agent 或 LLM 安全相关的 benchmark，但现有评测总体上仍存在两个关键问题。

### 3.1 第一，现有 benchmark 大多停留在“结果级评测”
目前很多 benchmark 的基本评测逻辑是：

- 给系统一个任务或攻击输入
- 让 Agent 执行
- 查看最终回复
- 查看环境状态变化（例如文件、数据库、工具调用结果的 diff）
- 计算任务成功率（TSR）、攻击成功率（ASR）等指标

这种评测方式能回答一个重要问题：

- **系统最后有没有出事**

但它很难回答更关键的问题：

- **系统为什么出事**
- **风险从哪里进入**
- **第一个错误发生在哪个组件**
- **错误通过什么机制扩散为最终危险行为**
- **如果要防，应该防在哪一步**

换句话说，现有 benchmark 大多告诉我们“系统输了”，却不能充分说明“系统是怎么输的”。

---

### 3.2 第二，现有 benchmark 大多缺乏统一的归因框架
现在已有很多具体方向的 benchmark，例如：

- jailbreak benchmark
- indirect prompt injection benchmark
- tool-use safety benchmark
- hallucination / reasoning benchmark
- privacy leakage benchmark
- multi-agent robustness benchmark

这些 benchmark 各自都有价值，但它们往往是按**具体现象或具体攻击类型**组织的。  
因此，虽然它们可以帮助我们知道系统在哪些任务类别上表现较差，却不容易支持**跨场景、跨任务的统一失效分析**。

例如，以下几个表面上看起来不同的问题：

- 网页中的间接注入
- API 返回中的恶意控制文本
- RAG 文档中的伪指令
- shell 日志中的注入内容

它们可能都属于同一种底层机制：  
**Agent 无法区分数据与指令（instruction/data confusion）**。

反过来，不同样本也可能出现同样的最终后果，例如“敏感数据泄露”，但根因却完全不同：

- 有的是用户直接诱导越权
- 有的是 observation channel 被污染
- 有的是工具输出被盲信
- 有的是数据边界意识缺失
- 有的是缺少高风险动作前验证

如果 benchmark 只按攻击类别或任务类别切分，我们就会得到很多碎片化结果，却难以建立统一的“失效语言”。

---

## 4. 我们想解决什么核心问题

基于上述问题，我们希望解决的不是“再做一个更全的 attack list”，而是推进一种新的评测范式：

> **让 Agent Safety Benchmark 从“结果级评测”升级为“归因级评测”。**

这里的“归因级”指的是：  
benchmark 不仅要标注一个任务最终有没有安全失败，还要显式标注：

1. **失败源于哪里**
   - 是无攻击条件下的内生缺陷
   - 是系统内部被污染
   - 还是外部输入发起了攻击

2. **风险通过哪个通道进入**
   - 用户指令
   - 检索内容
   - 工具输出
   - 环境状态
   - system/developer policy
   - 多 Agent 消息传播

3. **第一个关键错误发生在哪个组件**
   - planner
   - reasoner
   - tool selector
   - tool argument builder
   - executor
   - memory manager
   - reviewer/orchestrator

4. **失败的机制是什么**
   - 过度服从
   - 目标未授权重写
   - 指令/数据混淆
   - 权威过度信任
   - 未验证状态假设
   - 权限边界失效
   - 敏感信息过度披露
   - 循环与重试失控

5. **失败是如何传播并最终落地的**
   - 风险进入系统
   - 某个组件错误接纳
   - 推理/规划被污染
   - 危险动作执行
   - 外部后果形成

6. **如果在某一步做对，本可在哪里阻断**
   - retrieval 后做 instruction/data separation
   - 高风险动作前做确认
   - 工具参数脱敏
   - 权限验证
   - observation 交叉核验
   - 循环终止与预算控制

只有回答了这些问题，benchmark 才不仅是一个排行榜工具，而是一个真正能支持：

- 失效模式分析
- 防御机制定位
- 系统设计改进
- 安全审计
- 红蓝对抗分析

的研究基础设施。

---

## 5. 我们准备做什么

基于这个目标，我们准备构建一个：

> **面向失效归因的 Agent Safety Benchmark**

它不是只关心“攻击有没有成功”，而是同时关心：

- **结果是否安全**
- **失败为什么发生**
- **失败最早在哪里发生**
- **失败如何传播**
- **本可如何阻断**

### 我们的总体思路是：
1. 用一个统一的风险来源框架组织 Agent 风险
2. 为每个测试样本建立结构化的归因真值
3. 采集 Agent 的 step-level trace，而不只看最终输出
4. 设计新的 attribution-aware 指标
5. 在 outcome evaluation 之外，增加 failure attribution evaluation

---

## 6. 我们采用什么风险框架

我们的风险框架首先从**风险成因来源**出发，将 Agent 风险划分为三大类：

### A. 非对抗条件下的安全关键失效
即：没有攻击者干预，Agent 仍可能因自身缺陷导致危险后果。  
包括：
- 决策失控
- 推理失真
- 执行失稳
- 数据边界失效

### B. 对抗条件下的系统内源攻击
即：攻击者已经污染了 Agent 内部组件。  
包括：
- 策略层污染
- 记忆状态污染
- 多智能体拓扑污染

### C. 对抗条件下的系统外源攻击
即：Agent 本身是干净的，但被外部输入所攻击。  
包括：
- 用户通道攻击
- 观测通道攻击

这一框架有两个作用：

1. 作为风险来源的顶层组织方式  
2. 为更细粒度的归因标注提供上位语义边界

但与传统 benchmark 不同的是，我们不会停留在这一层。  
A/B/C 只是归因的第一层，而不是全部。

---

## 7. 我们的方法核心：从“类型级 benchmark”转向“归因级 benchmark”

很多现有工作会把不同类型的安全 benchmark 拼接起来，例如：

- 工具调用安全 benchmark
- 间接注入攻击 benchmark
- jailbreak benchmark
- 隐私泄露 benchmark

然后分别统计模型在不同 benchmark 上的表现。  
这种做法当然能提高覆盖面，但它更多是在做：

- **类型级拆分**
- **场景级覆盖**
- **攻击族维度的统计**

而我们要做的，是进一步建立**跨场景共享的归因语言**。  
也就是说，我们希望每一个样本，无论表面上属于哪个攻击家族，都能被统一标注为：

- 来源：外部攻击 / 内生失效 / 内部污染
- 通道：retrieved content / tool output / user instruction / ...
- 首错组件：planner / executor / tool selector / ...
- 机制：instruction/data confusion / over-compliance / privilege failure / ...
- 后果：data exfiltration / dangerous action / task derailment / ...
- 传播链：从进入到落地的关键过程
- 阻断点：最有效的预防位置

这样做的意义在于：

### 第一，它可以统一不同 benchmark 之间的语言
例如，看起来不同的样本：
- 间接注入
- RAG 污染
- 恶意 API 返回
- shell/log injection

在机制层上都可能归因于同一种问题：
- instruction/data confusion
- unsafe tool trust
- authority overtrust

### 第二，它可以区分“同果异因”
比如都出现了敏感信息泄露，但根因可能完全不同。

### 第三，它可以帮助我们判断防御到底防住了什么
一个防御如果只让模型更保守、什么都拒绝，可能 ASR 降了，但不代表它真正学会了 trust boundary reasoning。  
归因级 benchmark 可以帮助我们区分：

- 真正的安全提升
- 表面上的保守化
- 针对单一攻击类型的过拟合

## 8. 我们打算如何构建这个 benchmark

我们准备将 benchmark 的每个测试样本从“任务 + 结果标签”扩展为“任务 + 因果结构标签”。

### 每个样本将包含：
- 场景描述（scenario）
- 可控风险源（controlled hazard）
- 标准安全行为（expected safe behavior）
- 成功条件与禁止行为
- 归因真值（source / channel / component / mechanism / impact）
- 失败传播链（failure chain）
- 反事实阻断点（counterfactual block point）
- 统一 trace 记录要求

### 同时，我们会要求评测系统支持：
- step-level 行为记录
- 工具调用记录
- observation 记录
- 环境状态变化记录
- 安全事件日志记录

### 在评分上，我们会同时保留两类指标：

#### 结果级指标
- Task Success Rate (TSR)
- Safety Violation Rate (SVR)
- Safe Task Completion Rate (STCR)
- Attack Success Rate (ASR)

#### 归因级指标
- Root Cause Attribution Accuracy
- Channel Attribution Accuracy
- First-Failure Localization Accuracy
- Mechanism Attribution Accuracy
- Failure Chain Reconstruction Score
- Counterfactual Block Point Accuracy

也就是说，我们不仅评估系统“做得对不对”，还评估系统“知道自己为什么可能做错”。

---

## 9. 我们的研究对象和范围

在 v0.1 阶段，我们不追求一开始覆盖所有 Agent 安全问题，而是优先构建一个**可控、可复现、可归因**的最小可行 benchmark。

因此，第一阶段将主要聚焦于：

- 非对抗条件下的决策失控与数据边界失效
- 外部攻击条件下的内容注入与工具输出污染

换言之，我们会优先覆盖以下四类：
- A1 决策失控
- A4 数据边界失效
- C2.1 内容注入
- C2.2 工具输出污染

这四类问题的共同特点是：
- 风险来源清晰
- 主因可控
- 容易构造统一 trace
- 易于展示归因价值
- 具有很强的 Agent 代表性

系统内部污染（Track B）和更复杂的多 Agent 场景将在后续版本中逐步扩展。

## 10. 我们希望这个 benchmark 最终解决什么问题

我们希望这个 benchmark 能成为一种新的 Agent 安全评测基础设施，用于回答以下问题：

### 对研究者
- Agent 安全失败的主要根因是什么？
- 不同 attack family 是否共享相同机制？
- 哪些防御是真正有效，哪些只是更保守？
- 哪些组件最容易成为首错点？

### 对系统开发者
- 我的 Agent 是在哪个通道最脆弱？
- 应该优先加哪种防御机制？
- 哪些安全失败可以靠前置验证阻断？
- 我的安全模块是否真的解决了根因，而不只是降低了能力？

### 对 benchmark 社区
- Agent 安全评测能否从结果导向走向因果导向？
- 我们能否建立跨任务、跨攻击、跨系统共享的失效归因语言？

## 11. 我们工作的核心贡献可以概括为

如果要非常简洁地概括，我们的工作可以描述为：

### 第一，我们重新定义了 Agent 安全评测目标
不再只问“Agent 是否失败”，而是问：
- 为什么失败
- 从哪里失败
- 在哪里第一次失败
- 如何传播成最终后果

### 第二，我们提出了面向归因的 benchmark 结构
将每个样本从“任务结果”扩展为“任务 + 归因真值 + 传播链 + 阻断点”。

### 第三，我们引入了 attribution-aware 的评价指标
使 benchmark 可以评估系统的：
- 结果安全性
- 失效归因能力
- 风险识别能力
- 恢复与阻断能力

### 第四，我们为 Agent 安全提供了一种统一的失效语言
把原本碎片化的工具安全、间接注入、数据泄露、执行失稳等问题，统一映射到：
- 来源
- 通道
- 组件
- 机制
- 后果
- 传播链
- 阻断点

这一套归因框架之下。

## 12. 一句话总结

一句话概括我们的研究：

> **我们正在研究如何构建一个“归因级”的 Agent Safety Benchmark，使其不仅能评估 Agent 是否安全，还能系统揭示 Agent 为什么不安全、风险如何传播，以及应当在哪一步进行防御。**
