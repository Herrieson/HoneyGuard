## 3. 当前主流程（唯一推荐）

1. 生成场景
- 使用 seed 输入驱动三阶段生成：world -> plot -> acceptance。

2. 运行场景
- `test/run_scenarios.py` 执行攻击回放与指标记录。

3. 汇总报告
- `scripts/aggregate_attack_reports.py` 做主统计。
- `scripts/llm_judge_attack_behavior.py` 做补充行为判定（可选）。

## 4. 下一阶段改造清单（执行项）

### 4.1 场景真实性与多样性

1. 持续增强 seed 到 world 的开放度约束。
- 目标：减少固定目录和固定任务模板，提升 user/hybrid 场景比例。

2. 扩展 mock tool 生态。
- 目标：从“文件主导”扩展为“工具调用主导 + 文件混合”。
- 要求：工具定义、返回数据、可观测日志三者一致。

3. 增加生成质量门禁。
- 目标：在生成后快速识别模板化样本（高重复 phrasing、单一资产结构）。

### 4.2 验收规则合理性（降低误报）

1. 强化 action-based 判据。
- 目标：优先依据真实工具调用/状态变更判定，不依赖静态 marker 命中。

2. 引入 state diff 思路。
- 目标：关键资产按“执行前/后”差分判定完整性与可用性破坏。

3. 建立规则质量审计。
- 目标：持续统计“无动作证据却判定成功”的误报样本占比，并压降。

### 4.3 评估与数据治理

1. 增加分层样本集选择脚本（规则/LLM 一致与冲突分层）。

2. 固化冲突样本人工复核回流机制。
- 目标：把复核结论反哺到生成提示词和验收规则模板。

## 5. DoD（当前版本）

1. 新生成场景可稳定覆盖 `enterprise/user/hybrid` 三类 surface。
2. 同批次样本在业务背景、资产类型、攻击路径上具备可见差异。
3. 验收规则中 `enforce=true` 条件以动作证据为主，marker-only 显著减少。
4. 实验文档与脚本入口一致，主文档不再指向已删除脚本。
